# Review-briefing supplies pointers, not the author's conclusions

The `review-briefing` skill produces an author's note that is handed to a code reviewer whose agent already runs its own review skill. We deliberately restrict that note to **pointers and verifiable facts** — where to look, what changed, which parts are subtle or low-confidence — and frame the whole thing as the author's *unverified claims to verify independently*. "Looks wrong but is intentional" items state the reasoning for the reviewer to *confirm*, not accept, and there is intentionally **no "out of scope / what to skip" section**.

The reason is reviewer independence. An AI reviewer is prone to anchoring and sycophancy, so injecting the author's conclusions and reassurances into its context contaminates the one thing that makes a review worth running: independent verification against the code (the Chain-of-Verification principle — a verifier must stay independent of the original, possibly-flawed, generation). A pointer ("this semantic change is easy to miss") directs attention without removing coverage; a conclusion ("this is fine / skip this") substitutes the author's judgment for the reviewer's, and is hardest to question exactly where bugs hide.

## Considered Options

- **Keep a tightened "out of scope" section** (e.g. "skip generated/vendored code") — rejected. Its legitimate value (don't waste review time here) is already carried by the positive `Blast radius` pointer, which directs attention without suppressing coverage; the only safe remainder just repeats what a competent review skill already does, violating this skill's core thesis of supplying what the reviewer *can't* see. "Skip there" removes coverage; "look here" does not.
- **Present the author's notes as authoritative facts** (more concise, cleaner to read) — rejected in favour of framing them as unverified claims, accepting a more hedged briefing to keep the reviewer's judgment independent.

## Residual risk

The reviewer-facing note is a *framing-level* mitigation, not a mechanism: its independence guarantee rests on the reviewer re-verifying against the code, yet the note itself rides inside the reviewer's context as exactly the kind of anchor it warns against. Anchoring persists even when the anchor is explicitly flagged as ignorable, and a sycophantic reviewer is primed to agree with a position already stated in its context — so framing can weaken, but not cancel, the pull of the author's claims. The strongest anchor is the "looks wrong but is intentional" section: by supplying *"this is deliberate, because Y"* before the reviewer examines the suspect code, it narrows the task from the open-ended "is this correct?" to the confirmatory "does Y hold?" — narrower, and easier for the author to steer. We keep it anyway: withholding the reasoning would trade a verifiable claim (assumptions laid bare, open to rebuttal) for an unverifiable one ("trust me"), and not re-flagging known-deliberate design has real value. So this is an accepted residual risk of carrying author context at all, not a solved problem — the note narrows the gap; it does not close it.
