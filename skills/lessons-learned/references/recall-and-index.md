# Recall and Index Rules

Use this file as the detailed source of truth for lesson selection, `_index.md`
structure, recovery, and duplicate handling.

## Storage Layout

Store lessons under `docs/lessons/`:

```text
docs/lessons/
├── _index.md
├── api-timeout-retry-pattern.md
├── db-migration-run-order.md
└── null-check-before-save.md
```

## `_index.md` Format

If `_index.md` exists, it should use this structure:

```markdown
# Lessons Index

> Auto-generated by lessons-learned. Do not edit manually.

| Card | Scope | Tags | Date |
| --- | --- | --- | --- |
| `api-timeout-retry-pattern` | project | api,timeout,retry | 2026-03-09 |
```

Keep rows sorted by `Date` descending.

## Recall Selection

### Step 1: Identify current context

- extract task keywords
- determine one working scope: `project`, `module`, or `feature`

### Step 2: Recover the index if needed

If `docs/lessons/_index.md` is missing:

- if no lesson cards exist, treat this as first run and skip recall
- if lesson cards exist, rebuild `_index.md` from card frontmatter before recall

### Step 3: Rank candidates

Rank cards in this order:

1. tag match strength
2. scope match strength
3. `confidence` descending
4. `date` descending

Legacy fallback: if a card has no `confidence`, derive it from `source`:

- `user-correction` -> `0.7`
- `bug-fix` -> `0.5`
- `retrospective` -> `0.3`
- missing both -> `0.3`

## Confidence Initialization

When creating a new card, initialize `confidence` from `source`:

- `user-correction` -> `0.7`
- `bug-fix` -> `0.5`
- `retrospective` -> `0.3`

Increase by `+0.1` only when later evidence confirms the lesson was useful.
Never exceed `0.9`.

### Step 4: Load only a small set

- load `1-3` primary cards
- expand with up to `2` related cards
- never exceed `5` total loaded cards

Use [linking-heuristics.md](linking-heuristics.md) to decide whether related
cards qualify and how to rank them.

## Duplicate and Update Rules

Before creating a new card:

- look for an existing card with the same lesson meaning
- prefer updating that card over creating a near-duplicate
- preserve the existing semantic card ID when updating

Create a new card only when the new lesson has meaningfully different reuse
conditions or prevention guidance.

## Index Update Rules

Whenever a card is created or updated:

- upsert its row in `_index.md`
- keep the `Card`, `Scope`, `Tags`, and `Date` columns in sync with frontmatter
- keep the table sorted by newest date first

## Broken Related Links

Broken related links are non-blocking.

If a related target is missing:

1. remove the invalid entry
2. warn briefly in output
3. continue recall or capture
4. optionally backfill with a newly qualified target

## Validation Checklist

- `_index.md` exists when lessons exist, or is rebuilt before recall
- `_index.md` rows match card metadata
- ranking order matches this file
- new-card `confidence` values follow the source initialization rule
- recall loads no more than `5` total cards
- duplicate handling prefers update over semantic duplication
- related-card expansion follows `linking-heuristics.md`
